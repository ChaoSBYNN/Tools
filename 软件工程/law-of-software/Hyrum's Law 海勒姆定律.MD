# Hyrum's Law  海勒姆定律

> 当一个接口被足够数量的用户使用时，不论你在接口合约中做了什么承诺，都会有相应数量的情况出现，你会发现自己对某些人的期望是错误的。

* 随着用户数量的增加，系统做的所有事情都变成了依赖点。即使是意外的副作用或漏洞，也可能成为某人工作流程依赖的“功能”。
* 海勒姆定律警告维护者，任何更改都可能破坏某些东西。消费者可能以你意想不到的方式集成了你的 API，包括依赖时间、错误信息、格式化等。
* 你软件的实际约定不仅仅是官方的 API/规范;它是在实际环境中观察到的完全一致的行为。约定甚至可以是非正式的，比如用户习惯的界面。

## Overview  概述

海勒姆定律描述了一种现象：软件的文档界面与实现细节之间的界限在实际中变得模糊。无论你在 API 合同中承诺了什么，如果人们频繁使用你的系统，他们就会依赖于你从未正式支持过的行为。

随着时间推移，实现成为接口。每个可观察的特征、性能特征或错误代码都可能被某人假设。这极大限制了系统变更的自由度，因为即使是内部修改也可能破坏围绕旧行为编写代码的用户。

## Examples  示例

Microsoft Windows 存在许多第三方应用依赖的未公开行为和漏洞。当 Microsoft 在新版本中修复或更改这些行为时，这些应用就出现了故障。Microsoft 经常不得不恢复或维持这些奇怪的行为以确保兼容性。

某个库可能在文档中声明函数返回的是无序列表，但目前它返回的是排序列表。有些用户开始依赖它是否排序。如果库后来变成真正的无序，这些用户的代码就会损坏。这条法律在 API 版本管理中尤其重要：即使是小改动，从某些人的角度来看也可能构成破坏性变更。

## Origins  起源

该定律以软件工程师 Hyrum Wright 命名，他是在 2011-2012 年间在谷歌工作时表达了这一观察。Hyrum 注意到，即使是对核心 Google 库的更改，本应是隐形的，也常常会因为某个项目依赖于未被文档化的行为而导致项目失效。

他的同事泰特斯·温特斯（Titus Winters）在谷歌软件工程领域创造了“海勒姆定律”一词以示认可。
