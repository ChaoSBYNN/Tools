# Broken Windows Theory  破窗理论

> 不要留下未修复的破窗（糟糕的设计、错误的决定或差劲的代码）。

* 如果你对小的错误和糟糕的风格视而不见，人们会认为质量不重要，他们就会发布更混乱的代码。
* 一个干净、维护良好的代码库会鼓励工程师保持其干净，而一个混乱的代码库则会鼓励偷工减料和进一步的退化。
* 在问题还小的时候解决它们。重构破坏性的代码，更新过时的文档，以防止代码健康状况的持续下滑。

## Overview  概述

软件中的破窗理论由《实用程序员》一书推广开来。在一个城市里，一个未修复的破窗会传递出被忽视的信号并招致更多的破坏行为。同样在代码中，一个明显的错误或混乱的代码段若未被处理，会导致更多开发者绕过流程或引入更多的混乱。

其核心思想是，如果质量问题被忽视，它们会不断累积。例如，如果一个团队习惯性地忽视失败的测试或在构建过程中对代码检查/错误视而不见，开发者会得到一个信息：提交粗糙的工作是可以接受的。这会加速代码的衰败（也称为软件熵）。

相反地，如果团队能快速修复小问题并保持高标准，就能创造一种质量文化。

## Examples  例子

考虑一个存在一些明显拙劣修补的代码库，比如带有 `// TODO: fix this hack` 注释但从未被修复的函数。如果新成员看到这些，他们可能更倾向于添加自己的修补（“这个项目反正到处都是修补…”）。或者如果有一个没有测试的模块（从测试覆盖率的角度看是一个“破损的窗户”），随着纪律的侵蚀，其他模块也可能开始失去测试。

在另一边，想象一个项目，维护者会积极修复风格问题并在发现问题时简化代码。通过参与这样的项目，开发者可以快速学会匹配那种整洁。

许多团队通过偿还技术债务和清理代码扭转了项目。一旦事情变得完善，他们发现引入的新错误更少，因为每个人都更加小心。

## Origins  起源

破窗效应源自犯罪学（詹姆斯·Q·威尔逊和乔治·凯林，1982 年）。在软件领域，安迪·亨特和戴夫·托马斯在《实用程序员》（1999 年）中将其作为隐喻加以应用。他们主张及时修复小问题。

杰夫·阿特伍德和其他博客作者也在软件项目维护的背景下讨论了这个原则。

> Software Entropy  软件熵
> 代码随时间推移而日益退化并变得混乱的倾向也被称为“软件熵”现象。另一个解释代码混乱和缺乏整洁原因的相关概念是“破窗理论”。混乱的代码往往会变得更混乱，所以不要让第一扇窗户保持破碎。
