# The Law of Leaky Abstractions 漏水抽象定律

> 所有非平凡的抽象在某种程度上都存在泄漏。

* 无论设计得多好，抽象（库、框架等）都有依赖内部细节的边缘情况。
* 开发者应明白，使用高级工具并不能免除他们对底层发生了什么的了解，至少在基础层面是这样。
* “泄漏”意味着你可能会遇到性能问题、漏洞或行为，迫使你考虑抽象所依赖的底层系统（例如网络、操作系统）。
* 在创建抽象时，尽量减少泄漏，并记录可能出现的漏洞。

## Overview  概述

该定律指出，在复杂系统中，我们创建的抽象本应隐藏复杂性，但在某些情况下不可避免地失败 ，从而向程序员揭示了潜在的复杂性。

例如，考虑一个 ORM（对象关系映射器），它可以让你把数据库条目当作对象来处理。大多数时候它能正常工作，但偶尔会遇到性能糟糕的情况，需要考虑生成的 SQL 查询。这种抽象“泄露”了数据库的详细信息。

定律并没有说抽象是坏的。它们是必不可少的。但它提醒我们 ，必须为崩溃做好准备 。

## Examples  示例

一个很好的例子是高级语言中的内存管理 。像 Java 和 Python 这样的语言通过垃圾回收抽象了手动内存分配。然而泄漏仍然会发生（由于残留引用导致对象无法释放），或者你会遇到抽象的极限（例如 GC 暂停影响性能）。开发者突然需要了解垃圾回收内部的工作原理，而“无限内存”的抽象概念已经泄露。

网页开发者通常将 HTTP 请求视为快速处理，抽象网络。然而，当延迟激增或数据包丢弃时，网络的真实信息会泄漏到你的应用中。

## Origins  起源

乔尔·斯波尔斯基在 2002 年的一篇博客文章中提出了这项法律。他举了 TCP（抽象了基于 IP 的可靠连接，但在不良网络上抽象会泄漏，导致超时）或虚拟内存抽象等例子。

这个概念与许多早期计算机领域的思想产生共鸣，本质上是用抽象方式重新表达“没有免费午餐”。

> 所有模型都是错误的（乔治·博克斯定律）
> 乔治·博克斯说：“ 所有模型都是错误的，但有些是有用的。” 你构建的每一个抽象，无论是类层级结构、API 契约还是数据模型，都是谎言。它基本上是一个简化版的现实，忽略了那些不方便的细节 。问题不是你的模型是否错误。关键在于它是否在某些方面是错误的。
> 你的数据库模式假设用户只有一个邮箱地址，直到没有为止。你的付款模式将退款视为反向付款，直到退款单出现。接受每个型号都会泄漏，设计时要针对泄漏，而不是逆流而行。
